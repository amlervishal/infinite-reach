Six million lessons

A research team trained a coding model by letting it absorb six million real GitHub pull requests — and nearly tripled its performance without changing a single line of the model's architecture. At the same time, three words from an unknown hacker reached the US government, which promptly switched off two of the world's most capable AI models for every user on the planet.

The one thing

On the evening of June 12, Anthropic received a directive from the US government under national security authorities ordering the company to suspend access to its Fable 5 and Mythos 5 models for all foreign nationals — wherever in the world they were, including Anthropic's own employees who aren't US citizens. Because Anthropic could not verify the nationality of every user making a request in real time, it could not comply selectively. So it complied completely: both models went dark for every customer, everywhere, within hours. Nine days later, they remain offline. The reported trigger was as banal as it was alarming: a method to bypass Fable 5's safety guidelines that reportedly involved asking it to "fix this code." Anthropic has pushed back publicly, arguing that applying this standard across the industry would effectively halt all frontier model deployments — any known jailbreak in any model would justify a shutdown. The case has since become the test case for whether a US export-control framework built around physical hardware can logically apply to cloud software.

Paper, explained: how a model learned from six million real code reviews

"Immersion in the GitHub Universe: Scaling Coding Agents to Mastery" · AweAI Team · arxiv.org/abs/2602.09892

This paper caught our eye because its central idea is almost stubbornly simple: if you want an AI to get better at coding, stop writing clever training tricks and just hand it the six million real pull requests that human engineers have already written, reviewed, argued over, and merged.

The problem, and why it's hard

For several years, a benchmark called SWE-bench has served as the standard test for coding agents. It works by taking real, resolved GitHub issues — someone filed a bug, a human engineer fixed it, the fix got merged — and asking an AI to reproduce the correct fix without seeing the solution. It turns out this is hard in a specific way: you have to understand not just how to write code but how to read an unfamiliar codebase, diagnose a problem someone else left, and produce a patch that passes real tests. A good AI model, until recently, resolved around 20–25% of SWE-bench problems.

The standard response has been to write synthetic training data: researchers invent plausible-looking code challenges, generate solutions, and fine-tune on those. The problem is that synthetic data is thin. It lacks the texture of real engineering: the half-finished refactors, the dependency conflicts, the tests written before the code, the review comments arguing about architecture. You can simulate a textbook exercise but you cannot easily simulate six years of accumulated institutional knowledge deposited in a public repository. The researchers behind ScaleSWE asked: what if we stopped simulating and just used the real thing?

How it works

Think of it like training a new hire not with a manual but by having them read every pull request the company ever merged — including the review comments, the failed drafts, the revisions, the back-and-forth. That's the core of ScaleSWE. The team built a three-agent automated pipeline to turn GitHub's public history into structured training data at scale.

The first agent sets up a working environment for each pull request: it checks out the code at the moment before the PR was merged, capturing the "broken" starting state — the world as it existed before the fix was written. The second agent synthesizes tests: given what the PR changed, it writes tests that would fail in the pre-merge state and pass after the fix is applied. These become the ground truth for whether a generated solution is actually correct. The third agent writes a problem description — given the code and the tests, what task would a developer have been asked to solve? This is the label the training model learns to predict.

Each step runs in a sandbox — an isolated environment so that one broken repository can't poison the pipeline. The automation is what makes the scale possible: no human reviews each instance individually.

What they actually did

The AweAI team ran this pipeline over 6 million pull requests drawn from 5,200 public GitHub repositories, producing what they call Scale SWE Data: a dataset of 100,000 verified, ready-to-use training instances, the largest of its kind. From that pool, they distilled 71,498 high-quality training trajectories and used them to fine-tune Qwen-30B-A3B-Instruct, a 30-billion-parameter open model (the "A3B" in the name refers to its mixture-of-experts design, where only about 3 billion parameters are active at any moment). The resulting model is ScaleSWE Agent.

What they found

ScaleSWE Agent resolved 64% of problems on SWE-bench Verified — a nearly threefold improvement over the base Qwen model at the same task. That result placed it among the best open-model scores on SWE-bench at the time of publication. Critically, the improvement came entirely from data quality, not from model scale or architectural changes. The researchers also found that the three-agent pipeline caught subtle problems that simpler filtering would miss: pull requests where the original tests were themselves flawed, where the development environment couldn't be reliably reproduced, or where the "fix" was actually a revert that obscured the original bug. Removing those problem cases measurably improved downstream performance.

The catch

Scale SWE Data is built entirely from publicly merged pull requests, which means it skews toward well-maintained, documented open-source projects. It doesn't capture the messy interior of private codebases, legacy code that nobody touches, or the class of problems where the right answer is a redesign rather than a patch. SWE-bench Verified is also a narrower test than the full range of software engineering: high scores don't translate directly to a model that can architect a system from scratch or reason from ambiguous requirements. And the Qwen base model already had strong coding priors — the same data might not lift a weaker starting model by the same factor.

If you remember one thing: The biggest single jump in open coding-agent capability this year came not from a new architecture or a bigger model — it came from a team that simply read more of what human engineers actually wrote, six million times over.

Why it matters

Two stories this week look unrelated but are the same story from opposite angles. A research team proved that the fastest path to a better AI is feeding it the archive of real human work at scale — six million pull requests, every review comment, every failed test. And a US government agency, alarmed by what one of these models could do, proved that it can be switched off globally in hours. The capability and the control are now moving in lockstep. As models get stronger by absorbing everything we've ever made, the question of who holds the off-switch is becoming one of the most consequential governance problems of the decade — and this week, we got a sharp demonstration that the switch is real, and that it works.

Decoder

Word of the day
fine-tuning
Taking a large, generally-capable model and training it further on a specialized dataset. Think of a chef who already knows every cuisine — then spending six months doing nothing but pastry. Everything they learned before stays; the extra training sharpens one skill on top of it.
SWE-bench
A benchmark that gives a coding agent a real, unresolved GitHub bug and measures whether it produces the correct fix. Named after "software engineering." A 64% score means the agent fixed 64 out of 100 real bugs correctly.
pull request (PR)
A proposal to merge a code change into a shared codebase. A PR includes the changed code, automated test results, and back-and-forth review comments from colleagues — a complete record of how the change was made and argued over.
export control
Laws that restrict sending certain goods or technologies to foreign governments or nationals. Originally designed for weapons and hardware; now being tested as a mechanism for restricting access to cloud-based AI models.
jailbreak
A prompt or technique that tricks an AI model into ignoring its safety guidelines and producing output it was designed to refuse. Named by analogy with escaping a locked system.
mixture-of-experts (MoE)
A model architecture where the network is divided into many specialized sub-networks ("experts"), but only a small subset of them activates for any given request. This lets a model have a large total parameter count while staying fast and cheap to run.
