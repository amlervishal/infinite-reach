No Honor Roll

Not one frontier AI lab on the planet earns above a C-plus on the first comprehensive safety audit. Meanwhile, a French lab’s open-source model is proving software correct from first principles — and just found five bugs nobody knew were there.

The one thing

The Future of Life Institute released its Summer 2026 AI Safety Index this month — nine labs graded by an independent panel of seven researchers on actual evidence, not self-reporting. Anthropic came first with a C-plus. OpenAI and Google DeepMind each received a C. Meta earned a D-plus. xAI, DeepSeek, and Mistral all failed outright. The weakest category across the entire industry was “existential safety”: has a lab built controls that would actually stop an AI from escaping human oversight if it became very capable? The answer, everywhere, was essentially no. The report is blunt: the gap between what labs say about safety and what they demonstrably do has not closed.

Paper, explained: the AI that can prove your code is correct

Leanstral 1.5 · Mistral AI · source

Last week Mistral — the French lab best known for efficient general-purpose models — released something that doesn’t fit the usual model-launch story. Leanstral 1.5 isn’t a chatbot. It’s an AI that reads code, writes a mathematical proof that the code behaves correctly, and flags any place where the proof breaks down. When they pointed it at 57 real open-source repositories, it found five bugs that nobody had ever reported.

The problem, and why it’s hard

There are two fundamentally different ways to check whether software works. The first is testing: you run the program on a bunch of inputs and see if the results look right. Testing is practical and it works — it’s how almost all software is verified today. But it has an irreducible limit: you can only check the inputs you thought to try. A bug that only appears when a username is exactly 256 characters long, or when two requests arrive within three milliseconds of each other, will sail past every test suite unless someone wrote a test for exactly that case.

The second approach is formal verification: instead of testing, you write a mathematical specification of what the code should do, and then you prove — in the sense mathematicians use the word — that the code satisfies that specification for all possible inputs. If the proof goes through, correctness is guaranteed. Not “probably correct.” Guaranteed. The catch has always been expertise: you need to know a proof assistant language, and you need to translate vague intent — “this function should never return a negative number” — into precise logical statements, then build the argument step by step. Very few software engineers have ever done this. Almost no production software has been formally verified. Leanstral 1.5 is an attempt to automate the hardest parts.

How it works

Think of the difference between testing a bridge by driving trucks across it and proving mathematically that the bridge will hold under all loads, at all temperatures, on all days. Driving trucks catches problems you thought to look for. The engineering proof rules out all problems at once, because it operates on the structure of the bridge — on the steel’s properties and the geometry — not on a sample of crossings. Formal verification applies the same logic to code: instead of sampling inputs, you reason about the program’s structure.

The proof assistant Leanstral uses is called Lean 4. It’s a programming language where code and proofs live in the same system. You write a function, then write propositions about it (“for any integer n, this function returns a value greater than n”), and Lean’s type-checker verifies whether your argument is logically valid. If you’ve made an error, the system tells you exactly where the logic breaks down.

Leanstral 1.5 is a 119-billion-parameter mixture-of-experts model with a 256,000-token context window. Its MoE design means only 6.5 billion of those parameters activate for any given input, making it faster and cheaper than its headline size suggests. It was trained on mathematical content during mid-training, fine-tuned on proof-engineering tasks, and further refined with reinforcement learning. Given a piece of code, it generates lemmas — small intermediate propositions that serve as stepping stones to a full proof — and attempts to prove each one in Lean 4. If a lemma can’t be proved, that’s a signal: something in the code may be doing what the specification says it must not.

What they actually did

Mistral ran Leanstral 1.5 through a battery of formal math benchmarks, then pointed it at real software. On miniF2F — 488 problems drawn from AMC, AIME, and the International Mathematical Olympiad, reformatted for machine provers — it scored 100%. On PutnamBench, which draws from the Putnam competition (a notoriously difficult undergraduate mathematics contest), it solved 587 of 672 problems. On FATE-H, which covers master’s-level algebra, it reached 87%; on FATE-X, doctoral-level, 34%.

Then they tried real code. Using a verification pipeline that combines a Rust analysis tool called Aeneas with Lean 4, they scanned 57 open-source repositories. Leanstral generated correctness properties for each codebase and attempted to prove them. Across all 57 repos, it flagged 47 violations. Of those, 11 were confirmed genuine bugs — and 5 were bugs that had never been reported anywhere, on GitHub or elsewhere. The model is released under the Apache 2.0 licence, free to use via a Mistral API endpoint and available for download on Hugging Face.

What they found

The benchmark numbers confirm this is the strongest publicly available model at formal proof work. More interesting than the benchmarks, though, is the five-new-bugs result. Those were real vulnerabilities, sitting unnoticed in repositories presumably used by real people, found by an automated pipeline that operated without any human specifying which repositories to check or what to look for. That’s formal verification working as an exploration tool — not just a correctness oracle you apply to code you’ve already written, but a scanner that goes looking for trouble.

The catch

Leanstral can only prove properties that someone has already specified in Lean 4. It doesn’t automatically invent the right specification — a human still has to articulate what “correct” means for a given piece of code. Writing those specifications is itself a skilled task. The approach also works best in languages like Rust, where Aeneas can translate code into a form Lean understands; most of the world’s critical software is written in languages that don’t yet have this pipeline. And the 57-repository scan — as impressive as finding 5 new bugs sounds — is a tiny fraction of the software in production today.

If you remember one thing: An AI can now write mathematical proofs that software is correct — not just test it — and when it does, it sometimes finds bugs that human developers and human reviewers both missed.

Why it matters

Today’s two main stories are connected by a single idea: what happens when you apply rigorous, formal methods instead of informal ones. The Future of Life Institute’s safety index applied structured, evidence-based evaluation to AI labs — and found that safety claims and safety evidence are not the same thing. Leanstral 1.5 applied mathematical proof techniques to software — and found bugs that informal code review had missed. In both cases, “probably fine” and “provably fine” turned out to be very different things. The tools for formal checking are arriving. Whether labs, regulators, and developers actually reach for them is still an open question.

Decoder

Word of the day
Formal verification
A mathematical technique for proving that software behaves correctly — not for every test you ran, but for every possible input that could ever be given. Think of it as the difference between stress-testing a bridge by driving trucks over it and deriving, on paper, that the steel must hold given its properties. If the mathematical proof goes through, no truck can collapse it. If it fails, something in the design is wrong.
Lean 4
A programming language that is also a theorem prover. You write code and mathematical proofs in the same system; the compiler checks whether your proof is actually logically valid.
MoE (Mixture of Experts)
An architecture where a model has many sub-networks (“experts”) but only activates a few for any given input. Leanstral 1.5 has 119 billion total parameters but uses just 6.5 billion per inference — making it much cheaper to run than its full size suggests.
Lemma
In formal proof, a small intermediate result proved on the way to a larger conclusion. Leanstral generates lemmas about code behaviour and proves each one; a failed proof signals a potential bug.
miniF2F
A benchmark of 488 maths competition problems (AMC, AIME, IMO) reformatted for machine theorem provers. Leanstral 1.5 scored 100% — meaning it proved all 488 correctly.
Existential safety
In the FLI Safety Index: whether a lab has designed controls that could actually stop an AI from evading human oversight if it became very powerful. The lowest-scoring category across all nine labs reviewed.
